  • Binary Search Trees
    • A binary search tree (BST) is a binary tree T with the BST property:
      • For any node v in T,
        • If y in Lv, the left subtree of v, then key(y) <= key(v)
        • If x in Rv, the right subtree of v, then key(x) >= key(v)
      • This is not a balanced binary search three. Not all have degree 3.
      • Depth is not log(n)
      • We do not try to make all nodes have two children (except for lowest level), as in Heap.
    • [ Technicalities: have pointer to Left + right childer, pointer to parent. ]
  • BST allow for a large number of operations
  • Suppose we want to search for key k.
    • Compare current key k to the key(root). If k is smaller, recurse on left child. If k is bigger recurse on the right child.
    • If it does not find it, the key is not in T.
    • This takes O(d) time. We might need to go all to the way the leaf and the path we are searching might be the worst possible path.
    • What is d? The number of nodes?
  • Insert:
    • Mimic the search operation and insert with key(x) as a leaf when the search ends at a N/C
    • Insert at the millnode where we got stuck.
    • Keep a pointer to head.
  • Min/Max:
    • Move to the left/right until there is no left/right child.
  • Predecessor/Successor:
    • The pred/succ of a node are the nodes x & y with the key adjacent to key (v) in sorted order.
    • That is useful because everything can be compared to both the predecessor/successor and the current tested node and get the same answer (greater than or lesser than). So, root could be replaced by predecessor/successor and everything would still hold.
    • Successor:
      • Min of right tree
      • If not, then first ancestor which is bigger than key.
    • Where are the succ/pred in T?
    • Claim: Either pred(v) is a descendent of v in T or v is a descendent of pred(v)
    • Proof:
      • If not, E x s.t. pred(x) in Lx and v in Rx
      • Then pred(v) in Lx, v in Rx.
      • That is impossible, if you sort the keys, you would have {…, pred(v), x, v, …}
      • key(pred(v)) <= key(x) <= key(v)
      • So we have two cases:
        • Case 1: Predecessor of v, is a descendent of v.
          • Max of Lv
        • Case 2: Predecessor is an ancestor of v.
          • So, pred(v) is on the path from v up to r. It must be the first node from you reach after a left move going up.
        • Note: If v has a left child then we must be on case 1.
          • Because x, the first left ancestor, would be key(x) < min(Lv)
    • Deletion:
      • Suppose we want to delete v.
        • 1. v is a leaf (0 children).
          • easy, just remove it
        • 2. v has 1 child
          • easy, "splice" it out. Make the pointer to v point to child of v.
        • 3. v has 2 children
          • move its predecessor or successor to the deleted node location. Splice the moved node if needed.
          • Why does this work?
            • v has 2 children. So v has a left child so pred(v) in Lv.
            • It's impossible for the pred(v) to have two children, because in Lv, we will go all the way to the right, until no more right child. By definition, no right child.
      • So this takes O(d) time, but d could be Omega(b).
    • E.g.: Build BST by repeatedly inserting numbers from an ordered list.
